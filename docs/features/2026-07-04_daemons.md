# Daemons: managed long-running processes with model-in-the-loop log observation

## Introduction

Actions run to completion — a **daemon** is the sibling that is *supposed to keep running* (dev
server, compile watcher, test runner in watch mode). Its result is not an exit code but a
**status over time**. This feature adds daemons as a first-class concept: declarative
definitions, a supervised runtime, and **log observers** that watch the output stream (regex or
a cheap model) and turn what they see into events. The headline sink: **notifying the coding
agent working in the same worktree** by injecting a `[daemon:<name>]`-prefixed user message into
its stream-json session ("the dev server just crashed, here's the stacktrace").

Related/dependent plans:

- Builds directly on the [command registry](2026-06-30_command-registry.md): a running daemon
  **is** a registry command (new `CommandKind.DAEMON`, a PTY like an interactive action),
  reusing the PTY session, ring-buffer replay, terminal re-attach, per-line log persistence, and
  terminate machinery. Daemons add supervision *around* a `CommandSession`, not a second
  process subsystem.
- Mirrors the [actions](2026-05-01_actions.md) definition model: `AbstractDaemonDefinition`
  (`@MappedSuperclass`) → global `DaemonConfiguration` + repo-scoped `RepositoryDaemon`
  (cascade-deleted with the repo), same env element-collections, same
  `DaemonResolutionService.effectiveDaemons` global ∪ repository merge.
- The agent-notification sink targets sessions launched by the
  [coding-agent harness](2026-07-01_coding-agent-harness.md) and speaks the
  [stream-json chat](2026-07-01_stream-json-chat.md) protocol — the injection point is
  `CommandRegistry.chatSend`, so an injected event rides the unified line stream: it appears in
  the live transcript and persists for replay with zero extra wiring
  ([persistent-chat-sessions](2026-07-04_persistent-chat-sessions.md)).
- Complements the [worktree chat dialog](2026-07-04_worktree-chat-dialog.md): daemon events
  surfacing inside that chat is the payoff of both features combined.
- [Observability](../feature-ideas/observability.md) builds on this launch path (an `otel`
  toggle would inject `OTEL_EXPORTER_OTLP_*` into the daemon's environment).

## What was built

### Domain model (`domain.daemon`, new BCE area)

- `AbstractDaemonDefinition` (`@MappedSuperclass`): `name`, `description`, `startScript`
  (4000, runs verbatim in the worktree, stays in the foreground), `readyPattern` (nullable
  regex; first output match flips STARTING → READY), `stopSignal` (normalized signal name,
  default `TERM`), `restartPolicy` (`NEVER`/`ON_FAILURE`/`ALWAYS`) + `maxRestarts`,
  timestamps. Subclasses `DaemonConfiguration` (global) and `RepositoryDaemon` (repo-owned,
  cascade-deleted) each add their own `environment` map and ordered `observers` list
  (`LogObserver` `@Embeddable`: kind `PATTERN`/`MODEL`, regex + severity, or classifier prompt
  override). Migration `V14__daemon.sql` (also extends the `command.kind` check with `DAEMON`
  by recreating the column — the V13 check was anonymous).
- **No instance entity.** The durable record of a run is its `Command` row (kind `DAEMON`, one
  row per (re)launch); supervisor state lives in-memory next to the registry's sessions. A JVM
  restart reconciles daemon commands to INTERRUPTED like any command; instances read STOPPED.
- Regexes and signals are validated at definition time (`DaemonDefinitionValidator`) so a broken
  pattern fails the request, not the reader thread.
- `DaemonSeeder` seeds one idempotent demo daemon ("Python HTTP server", ready pattern
  `Serving HTTP`, one PATTERN + one MODEL observer).

### Runtime: `DaemonSupervisor`

`start(repoId, worktreeId, daemonId)` resolves the definition, launches via
`CommandService.launchDaemon` (PTY spawn with observer sinks attached *before* first output),
and drives the state machine `STARTING → READY → DEGRADED/CRASHED/RESTARTING → STOPPED`:

- READY via the ready-pattern sink, or after a grace period (`qits.daemons.ready-grace-ms`,
  default 10s) when no pattern is set.
- On exit: `restartPolicy` decides — relaunch (new `Command` row, incremented restart count,
  exponential backoff from `qits.daemons.restart-backoff-initial-ms` doubling to a 30s cap) or
  settle in CRASHED (restarts exhausted / NEVER with non-zero exit); a clean exit settles
  STOPPED. Explicit stop and manual terminate always beat the policy.
- `stop()` sends the definition's stop signal to the **process group**
  (`CommandRegistry.signal` → `kill -s <SIG> -- -<pid>`; the PTY leader is a session leader),
  waits `qits.daemons.stop-grace-ms` (default 5s), then falls back to the registry's
  `terminate()`. Stopping is explicit — the no-auto-cleanup rule holds.
- **Singleton per (worktree, daemon)** enforced: a second `start` is a 400.
- A `TailSink` keeps the last ~40 output lines so crash events carry the evidence without a DB
  read on the exit path.
- Because the instance is a registry command, the existing terminal socket gives log tailing
  and re-attach in the UI for free.

### Log observers

Observers are `CommandOutputSink`s (the registry's existing fan-out); `LineFramingSink`
reassembles PTY chunks into ANSI-stripped lines:

- **PATTERN**: regex per line → `ERROR_DETECTED` finding with the configured severity and the
  line as evidence. Throttled (≥10s between findings per observer) so a crash loop can't flood
  the feed or the agent's chat.
- **MODEL**: buffers lines, flushes on a ~2s idle debounce or 16KB, and a case-insensitive
  `error|exception|warn|fail|fatal|panic|traceback` pre-filter gates the model call entirely —
  quiet, healthy logs cost zero. One call per batch against the Messages API (Haiku,
  `qits.daemons.classifier-model`), with a long stable system prompt marked
  `cache_control: ephemeral` (≥1,024 tokens, so steady logging pays mostly the cached rate).
  Strict one-line output contract `NONE` or `SEVERITY|error-type|summary|first-line-offset`,
  parsed leniently; the offset picks where the event's excerpt starts.
- **Deviation from the idea doc:** the client is a small hand-rolled `HttpClient` + Jackson
  wrapper (`AnthropicLogClassifier`) behind a `LogClassifier` interface, not the `com.anthropic`
  SDK — one endpoint didn't justify the dependency, tests fake the interface, and the interface
  keeps a later SDK swap trivial. The rejected alternative (shelling out to `claude -p` per
  batch) stays rejected. Key via `qits.anthropic.api-key` (defaults from `ANTHROPIC_API_KEY`);
  **without a key MODEL observers are inert** and PATTERN observers still work.

### Events and sinks

Every transition and finding is a `DaemonEventDto` published through `DaemonEventService`
(bounded in-memory ring for the UI feed). Everything above INFO severity is forwarded to
`DaemonAgentNotifier`:

- Resolve **the newest running CHAT command in the same worktree** — the server-side twin of
  the frontend's `newestRunningChat` rule (`CommandRepository.findRunningByKindAndWorktree` +
  registry liveness) — and `chatSend` the formatted message
  (`[daemon:<name>] SEVERITY: summary` + fenced log excerpt). It hits Claude's stdin as a real
  user turn and the transcript/persistence via the normal unified stream. Mid-turn injection is
  fine: the CLI queues stdin messages and picks them up next turn.
- **No chat running** → the message is spooled per worktree (`DaemonEventSpool`, bounded);
  `AgentLaunchService.launchChat` drains it into every new chat right after the seed prompt.
- Severity mapping keeps the agent's signal clean: CRASHED → ERROR, RESTARTING → WARNING,
  ERROR_DETECTED → the observer's severity; READY/STOPPED/DEGRADED transitions are INFO
  (DEGRADED is INFO because the ERROR_DETECTED that caused it already notified).
- **Every stream-json chat now launches with `--include-hook-events`** (`ClaudeCodeAgent.chat()`),
  giving the stream turn-boundary hook events for future busy/idle detection and better-timed
  injection.

### REST + UI

- `/api/daemon-configurations` (global CRUD) and `/api/repositories/{id}/daemons` (repo CRUD;
  net-new REST — actions only ever got MCP tools for this scope), nested-record controllers.
- `/api/repositories/{repoId}/worktrees/{worktreeId}/daemons`: GET effective daemons with
  status/restartCount/commandId (all of them, running or not — the everything-visible
  convention), POST `{daemonId}/start` / `{daemonId}/stop`, GET `events`.
- Worktree detail gains a **daemons panel** (`WorktreeDaemonsComponent`): status chips
  (severity-colored, restart count, pulse while starting), start/stop, "Logs" re-attaching to
  the instance's command terminal, and the recent-events feed (severity dot, expandable
  excerpt). Polls at 3s/5s.
- Definitions get a global library UI (`/daemon-configurations`: list, create/edit form with
  observer and env editors, delete) and a "Daemons" nav entry. Per-repo daemon *management UI*
  is deferred; the REST surface exists and repo daemons show up in the worktree panel.

## Decisions on the idea's open questions

- **Instance uniqueness**: singleton per (worktree, daemon) enforced.
- **API key**: one global config property; no per-day token cap yet (the pre-filter + debounce
  keep cost negligible; revisit if MODEL observers multiply).
- **DEGRADED**: does *not* auto-recover — reset only by restart or stop (simplest defensible
  rule; revisit with real usage).
- **Area**: new `domain.daemon` BCE area depending on `command`, mirroring how `agent` uses the
  registry without living in it.

## Explicitly deferred (unchanged from the idea)

- Feature-flow integration (phase gate on "dev server READY").
- Auto-spawned headless triage agent on `ERROR_DETECTED`.
- Hook-based injection into PTY-interactive agent sessions (stream-json chats are the target).
- Port management — parallel worktrees running the same daemon collide on hardcoded ports;
  documented limitation.
- Push/desktop notifications for the human; the UI feed covers iteration one.
- Observers on non-daemon commands (the sink mechanism is deliberately generic).
- Per-repo daemon management UI (REST exists).

## Verification

- `DaemonSupervisorTest` (domain, real processes in a cloned-fixture worktree): ready-pattern
  flip, ON_FAILURE relaunch with a new `Command` row + CRASHED settling with the output tail as
  evidence + agent message spooled, NEVER → CRASHED, clean exit → STOPPED, stop beats ALWAYS,
  singleton enforcement.
- `ObserverSinkTest`: chunk/ANSI line framing, PATTERN match + throttle, MODEL gating (quiet
  logs → zero classifier calls; error batch → one call → finding with offset excerpt; no key →
  inert), classifier reply parsing.
- `DaemonAgentNotifierTest`: an event lands as one `[daemon:…]`-prefixed stream-json user
  message on a running chat (visible in the persisted transcript); no chat → spooled, drained
  once.
- `DaemonConfigurationControllerTest`: CRUD round-trip incl. observers/env, regex/observer
  validation, seeded demo daemon present. `OpenApiSchemaExportTest` regenerated
  `docs/openapi.yml`; `pnpm generate:api` regenerated the client.
- `worktree-daemons.component.spec.ts`: chips + start/stop wiring + events feed rendering.
- Manual end-to-end against the packaged app: seeded daemon started in a real worktree →
  READY via pattern (and `curl :8000` served 200); external `kill -9` → RESTARTING event (exit
  137, excerpt attached) → automatic relaunch as a new command → READY again; graceful stop →
  SIGTERM (exit 143) → STOPPED with no policy-triggered relaunch; three DAEMON command rows.
