# Actions

## Introduction

> **Merged into Feature Flows.** The standalone `action` domain package was merged into `featureflow` as part of [`2026-05-01_feature-flows.md`](2026-05-01_feature-flows.md). Action configurations are now a subordinate part of the feature-flow domain rather than a standalone package.

> **Scope unified (2026-07-09).** The interim two-table split (global `ActionConfiguration` vs `repository_action`, introduced with repository-scoped actions) was folded back into the single `ActionConfiguration` table with a nullable `repository_id` (`null` = global) — see [`2026-07-09_unified-action-scope.md`](2026-07-09_unified-action-scope.md). The model below still matches, plus that FK.

This document describes the **action-configuration** entity — a shell script definition with an accompanying check script that determines whether execution is needed.

### Related / Dependent Plans
- [`2026-05-01_feature-flows.md`](2026-05-01_feature-flows.md) — Action configurations are linked to feature-flow phases via `FeatureFlowPhaseAction`.
- **Action execution & logging** (future): Running action-configurations and tracking their execution history is out of scope. It requires a missing runtime/primitive that does not yet exist.

## Goals

1. Define executable action-configurations that can be reused across feature-flow phases.
2. Allow action-configurations to self-report their necessity via a check script contract.

## Non-Goals

1. Direct database relationships to workspaces (join table is out of scope).
2. Executing action-configurations or tracking execution history.
3. Automatic scheduling or triggering of action-configurations.

## Domain Model

### ActionConfiguration

Represents a single executable task definition.

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID / ULID | Primary key |
| `name` | string | Human-readable name |
| `description` | string (optional) | What the action does |
| `execute_script` | string | Shell script to run when the action-configuration is executed |
| `check_script` | string | Shell script that determines if the action-configuration should run |
| `created_at` | timestamp | |
| `updated_at` | timestamp | |

## Check Script Contract

The `check_script` of an action-configuration must output a single line to `stdout` containing one of the following values:

| Value | Meaning |
|-------|---------|
| `required` | The action must be executed |
| `suggested` | The action should probably be executed |
| `optional` | The action may be executed |
| `unnecessary` | The action does not need to be executed |

Any other output or a non-zero exit code should be treated as an error state.
