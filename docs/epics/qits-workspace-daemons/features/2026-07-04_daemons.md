# Daemons: managed long-running processes with log observation

## Introduction

Actions run to completion — a **daemon** is the sibling that is *supposed to keep running* (dev
server, compile watcher, test runner in watch mode). Its result is not an exit code but a
**status over time**. This feature adds daemons as a first-class concept: declarative
definitions, a supervised runtime, and **log observers** that watch the output stream (a
per-line regex, or local classification off the severity vocabulary logs already carry) and turn
what they see into events. The headline sink: **notifying the coding agent working in the same
workspace** by injecting a `[daemon:<name>]`-prefixed user message into its stream-json session
("the dev server just crashed, here's the stacktrace").

> The original idea included a model-backed (LLM) observer making Anthropic API calls per log
> batch. That was built and then **deliberately removed**: qits makes no outbound API calls from
> log observation. The `LogClassifier` interface remains the extension point should a
> model-backed classifier return as a future feature.

Related/dependent plans:

- Builds directly on the [command registry](../../qits-workspace-commands/features/2026-06-30_command-registry.md): a running daemon
  **is** a registry command (new `CommandKind.DAEMON`, a PTY like an interactive action),
  reusing the PTY session, ring-buffer replay, terminal re-attach, per-line log persistence, and
  terminate machinery. Daemons add supervision *around* a `CommandSession`, not a second
  process subsystem.
- Mirrors the [actions](../../qits-feature-flows/features/2026-05-01_actions.md) definition model *in shape, not in scope*: a
  daemon is essentially a long-running, non-interactive action, but it is **repository-scoped
  only** — `RepositoryDaemon`, cascade-deleted with its repo, with the same env
  element-collections. The initially built global `DaemonConfiguration` scope (mirroring the
  action library) was a **mistaken specification and was removed** (`V19`): the dev server you
  test against belongs to a concrete repository; daemons are not a top-level concept beyond
  read contexts (the command registry their instances live in, the durable event feed).
- The agent-notification sink targets sessions launched by the
  [coding-agent harness](../../qits-coding-agents/features/2026-07-01_coding-agent-harness.md) and speaks the
  [stream-json chat](../../qits-coding-agents/features/2026-07-01_stream-json-chat.md) protocol — the injection point is
  `CommandRegistry.chatSend`, so an injected event rides the unified line stream: it appears in
  the live transcript and persists for replay with zero extra wiring
  ([persistent-chat-sessions](../../qits-coding-agents/features/2026-07-04_persistent-chat-sessions.md)).
- Complements the [workspace chat dialog](../../qits-workspace-detail/features/2026-07-04_workspace-chat-dialog.md): daemon events
  surfacing inside that chat is the payoff of both features combined.
- [Observability](../../qits-observability/features/2026-07-04_observability.md) builds on this launch path (an `otel`
  toggle would inject `OTEL_EXPORTER_OTLP_*` into the daemon's environment).
- [Workspace containers](../../qits-workspaces/features/2026-07-04_workspace-containers.md) relocates all
  registry execution — daemons included — into per-workspace containers, resolving the port
  limitation deferred below.

## What was built

### Domain model (`domain.daemon`, new BCE area)

- `RepositoryDaemon` (repo-owned, cascade-deleted — the **only** definition scope): `name`,
  `description`, `startScript` (4000, runs verbatim in the workspace, stays in the foreground),
  `readyPattern` (nullable regex; first output match flips STARTING → READY), `stopSignal`
  (normalized signal name, default `TERM`), `restartPolicy` (`NEVER`/`ON_FAILURE`/`ALWAYS`) +
  `maxRestarts`, timestamps, an `environment` map and ordered `observers` list (`LogObserver`
  `@Embeddable`: kind `PATTERN`/`LOG_LEVEL`, regex + severity for PATTERN). Migrations
  `V14__daemon.sql` (also extends the `command.kind` check with `DAEMON` by recreating the
  column — the V13 check was anonymous) and `V15__log_level_observer.sql` originally created a
  global `daemon_configuration` twin as well; `V19__drop_global_daemon_scope.sql` dropped it
  (see the correction note above), collapsing the `AbstractDaemonDefinition` mapped superclass
  and the `DaemonResolutionService` merge into the single entity and
  `RepositoryDaemonService.resolve`/`resolveAll`.
- **No instance entity.** The durable record of a run is its `Command` row (kind `DAEMON`, one
  row per (re)launch); supervisor state lives in-memory next to the registry's sessions. A JVM
  restart reconciles daemon commands to INTERRUPTED like any command; instances read STOPPED.
- Regexes and signals are validated at definition time (`DaemonDefinitionValidator`) so a broken
  pattern fails the request, not the reader thread.
- The cli `seed` command seeds one demo daemon ("Python HTTP server", ready pattern
  `Serving HTTP`, one LOG_LEVEL observer) on the demo repository (originally a startup-seeded
  global `DaemonSeeder`; with the global scope gone, seeding needs a repository to hang the
  daemon on).

### Runtime: `DaemonSupervisor`

`start(repoId, workspaceId, daemonId)` resolves the definition, launches via
`CommandService.launchDaemon` (PTY spawn with observer sinks attached *before* first output),
and drives the state machine `STARTING → READY → DEGRADED/CRASHED/RESTARTING → STOPPED`:

- READY via the ready-pattern sink, or after a grace period (`qits.daemons.ready-grace-ms`,
  default 10s) when no pattern is set.
- On exit: `restartPolicy` decides — relaunch (new `Command` row, incremented restart count,
  exponential backoff from `qits.daemons.restart-backoff-initial-ms` doubling to a 30s cap) or
  settle in CRASHED (restarts exhausted / NEVER with non-zero exit); a clean exit settles
  STOPPED. Explicit stop and manual terminate always beat the policy.
- `stop()` sends the definition's stop signal to the **process group**
  (`CommandRegistry.signal` → `kill -s <SIG> -- -<pid>`; the PTY leader is a session leader),
  waits `qits.daemons.stop-grace-ms` (default 5s), then falls back to the registry's
  `terminate()`. Stopping is explicit — the no-auto-cleanup rule holds.
- **Singleton per (workspace, daemon)** enforced: a second `start` is a 400.
- A `TailSink` keeps the last ~40 output lines so crash events carry the evidence without a DB
  read on the exit path.
- Because the instance is a registry command, the existing terminal socket gives log tailing
  and re-attach in the UI for free.

### Log observers

Observers are `CommandOutputSink`s (the registry's existing fan-out); `LineFramingSink`
reassembles PTY chunks into ANSI-stripped lines:

- **PATTERN**: regex per line → `ERROR_DETECTED` finding with the configured severity and the
  line as evidence. Throttled (≥10s between findings per observer) so a crash loop can't flood
  the feed or the agent's chat.
- **LOG_LEVEL**: buffers lines, flushes on a ~2s idle debounce or 16KB, and classifies the batch
  **locally** via `LogLevelClassifier` — the severity vocabulary logs already carry: level
  tokens (`FATAL`/`ERROR`/`SEVERE`/`PANIC`/`CRITICAL` → ERROR; `WARN`/`WARNING`/deprecations →
  WARNING), exception class names (`\w+Exception`/`\w+Error`, which also become the finding's
  `errorType`), and the Python traceback opener. Word-bounded so "Found 0 errors" stays quiet;
  routine INFO/DEBUG output is deliberately not a finding (the full stream is persisted as the
  command's log regardless — observers exist to surface problems). One finding per burst, with
  the excerpt starting at the classified offset; same ≥10s throttle as PATTERN. No
  configuration, no API calls, no cost.
- The classification seam is the `LogClassifier` interface (`classify(batch)` →
  `Classification(severity, errorType, summary, firstLineOffset)`); `LogLevelClassifier` is the
  only implementation. A future model-backed classifier would implement the same interface —
  explicitly out of scope now: **no outbound API calls from log observation**.

### Events and sinks

Every transition and finding is a `DaemonEventDto` published through `DaemonEventService`
(bounded in-memory ring for the UI feed). Everything above INFO severity is forwarded to
`DaemonAgentNotifier`:

- Resolve **the newest running CHAT command in the same workspace** — the server-side twin of
  the frontend's `newestRunningChat` rule (`CommandRepository.findRunningByKindAndWorkspace` +
  registry liveness) — and `chatSend` the formatted message
  (`[daemon:<name>] SEVERITY: summary` + fenced log excerpt). It hits Claude's stdin as a real
  user turn and the transcript/persistence via the normal unified stream. Mid-turn injection is
  fine: the CLI queues stdin messages and picks them up next turn.
- **No chat running** → the message is spooled per workspace (`DaemonEventSpool`, bounded);
  `AgentLaunchService.launchChat` drains it into every new chat right after the seed prompt.
- Severity mapping keeps the agent's signal clean: CRASHED → ERROR, RESTARTING → WARNING,
  ERROR_DETECTED → the observer's severity; READY/STOPPED/DEGRADED transitions are INFO
  (DEGRADED is INFO because the ERROR_DETECTED that caused it already notified).
- **Every stream-json chat now launches with `--include-hook-events`** (`ClaudeCodeAgent.chat()`),
  giving the stream turn-boundary hook events for future busy/idle detection and better-timed
  injection.

### REST + MCP + UI

- `/api/repositories/{id}/daemons` (repo CRUD, nested-record controller) is the whole REST
  management surface. The original `/api/daemon-configurations` global CRUD and its library UI
  (`/daemon-configurations` + nav entry) were **removed with the global scope**.
- **Daemon management lives on the repository MCP server** (`DaemonMcpTools`, mounted with
  `RepositoryMcpTools` at `/mcp/repository`, same `ProjectScopeGuard` project/repository
  scoping): `listDaemons`/`createDaemon`/`updateDaemon`/`deleteDaemon` (structured observer,
  source and env arguments) plus the runtime `listWorkspaceDaemons`/`startDaemon`/`stopDaemon`.
  Unlike actions — whose global library is managed on the separate "actions" server — a daemon
  is repository-owned configuration, so configuring it is part of working in that repository.
  The repository detail page's **"Configure with Claude"** session (`AgentMcpScope.ACTIONS`)
  therefore attaches *both* servers: the actions server for the action library plus the
  repository server narrowed to the repo, so one session can configure actions and daemons
  alike. The discovery catalogue (`listContextServers`) advertises the daemon tools on the
  repository server.
- `/api/repositories/{repoId}/workspaces/{workspaceId}/daemons`: GET the repository's daemons
  with status/restartCount/commandId (all of them, running or not — the everything-visible
  convention), POST `{daemonId}/start` / `{daemonId}/stop`, GET `events`.
- Workspace detail gains a **daemons panel** (`WorkspaceDaemonsComponent`): status chips
  (severity-colored, restart count, pulse while starting), start/stop, "Logs" re-attaching to
  the instance's command terminal, and the recent-events feed (severity dot, expandable
  excerpt). Polls at 3s/5s.
- Per-repo daemon *management UI* lives as subroutes of the repository detail page
  (`/repositories/{repoId}/daemons` list + `new` / `:daemonId/edit` form with observer, source
  and env editors — the former global library UI, rebuilt repo-scoped), reachable via a
  "Daemons" action on the repository detail page.

## Decisions on the idea's open questions

- **Instance uniqueness**: singleton per (workspace, daemon) enforced.
- **API key / token budget**: moot — the model-backed observer was removed; classification is
  local (`V15__log_level_observer.sql` converted the MODEL kind to LOG_LEVEL and dropped the
  per-observer prompt override).
- **DEGRADED**: does *not* auto-recover — reset only by restart or stop (simplest defensible
  rule; revisit with real usage).
- **Area**: new `domain.daemon` BCE area depending on `command`, mirroring how `agent` uses the
  registry without living in it.
- **Scope** (same-day correction): repository-only. The initially shipped global
  `DaemonConfiguration` library mirrored actions too literally — a daemon is defined on the
  repository whose dev server it is, and its management belongs to the repository MCP server
  where the agent already works with that repository.

## Explicitly deferred (unchanged from the idea)

- Feature-flow integration (phase gate on "dev server READY").
- Auto-spawned headless triage agent on `ERROR_DETECTED`.
- Hook-based injection into PTY-interactive agent sessions (stream-json chats are the target).
- Port management — parallel workspaces running the same daemon collide on hardcoded ports;
  documented limitation. Planned resolution:
  [workspace containers](../../qits-workspaces/features/2026-07-04_workspace-containers.md) — per-container network
  namespaces make the hardcoded port correct for every workspace simultaneously, no
  auto-allocation needed.
- Push/desktop notifications for the human; the UI feed covers iteration one.
- Observers on non-daemon commands (the sink mechanism is deliberately generic).
- ~~Log sources beyond the process output (file tails), durable events, per-line severity, and
  event↔source correlation~~ — implemented as
  [daemon-log-observation-expansion](2026-07-04_daemon-log-observation-expansion.md): FILE
  sources tailed into the same observers, `daemon_event` rows replacing the in-memory ring,
  source-qualified anchors, and severity-stamped log lines.

## Verification

- `DaemonSupervisorTest` (domain, real processes in a cloned-fixture workspace): ready-pattern
  flip, ON_FAILURE relaunch with a new `Command` row + CRASHED settling with the output tail as
  evidence + agent message spooled, NEVER → CRASHED, clean exit → STOPPED, stop beats ALWAYS,
  singleton enforcement.
- `ObserverSinkTest`: chunk/ANSI line framing, PATTERN match + throttle, LOG_LEVEL batching
  (quiet logs → no finding; error batch → one finding with the offset-based excerpt), and the
  local classifier's vocabulary (exception names as errorType, ERROR beats WARNING, "0 errors"
  stays quiet).
- `DaemonAgentNotifierTest`: an event lands as one `[daemon:…]`-prefixed stream-json user
  message on a running chat (visible in the persisted transcript); no chat → spooled, drained
  once.
- `RepositoryDaemonControllerTest`: CRUD round-trip incl. observers/env, regex/observer
  validation, cross-repository ownership enforcement. `DaemonMcpToolsTest`: full
  define→list→edit→delete lifecycle and start/stop through the repository MCP server incl.
  structured observer/source arguments, project-scope refusal, readable enum errors;
  `RepositoryMcpToolsTest` pins the server's exact tool surface incl. the daemon tools.
  `OpenApiSchemaExportTest` regenerated `docs/openapi.yml`; `pnpm generate:api` regenerated the
  client.
- `workspace-daemons.component.spec.ts`: chips + start/stop wiring + events feed rendering.
- Manual end-to-end against the packaged app: seeded daemon started in a real workspace →
  READY via pattern (and `curl :8000` served 200); external `kill -9` → RESTARTING event (exit
  137, excerpt attached) → automatic relaunch as a new command → READY again; graceful stop →
  SIGTERM (exit 143) → STOPPED with no policy-triggered relaunch; three DAEMON command rows.
