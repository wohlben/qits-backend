# "Create workspace" button for a workspaceless branch (adopt an existing branch)

## Introduction

On the repository-detail branch list, every branch that has **no matching workspace** now shows a **"Create workspace"** button. Clicking it creates a workspace that **adopts the existing branch** in place (it does not fork a new one), with `parent` set to the repository's configured main branch. One click, no dialog — the workspace id is derived from the branch name.

Related / dependent plans:

- `docs/epics/qits-workspaces/features/2026-07-04_workspace-containers.md` — the workspace = "branch ref in the bare origin + lazily-provisioned container" model that adoption slots into (the container still materializes lazily on first use).
- `docs/epics/qits-workspaces/features/2026-07-08_lazy-workspace-container-provisioning.md` — creation writes only durable state; adoption is the same **minus** the ref creation, since the ref already exists.
- `docs/epics/qits-workspaces/features/2026-07-07_workspace-sse-live-updates.md` — the new workspace rides the existing `invalidateRepository` live-refresh path.

**Not changed:** the "Integrate" action stays where it is (the per-node commits popover, Forward tab). It already falls back to the configured main branch for workspaceless branches (`openIntegrate` → `workspace?.parent ?? mainBranch()`; `WorkspaceService.mergeBranch` defaults a blank target to `repo.mainBranch`), so no work was needed there.

## What was built

### Backend — adopt an existing branch

`WorkspaceService.createWorkspace` gained a 6-arg overload with a `boolean adoptExisting` (the 4- and 5-arg forms delegate with `false`, so existing callers are untouched):

- New probe `branchExistsOnOrigin(originPath, branch)` — `git show-ref --verify --quiet refs/heads/<branch>` via `execAllowNonZero` (a non-zero exit is the "absent" answer, not a failure).
- When `adoptExisting` is set **and** the branch already exists, the origin-side `git branch` ref creation is skipped and the row is simply recorded over the existing branch. Otherwise the normal path creates the ref — and still **errors loudly if the branch already exists**, so a typo'd "branch off" name is never silently adopted.

`WorkspaceController.CreateWorkspaceRequest` gained the `adoptExisting` field, plus a backward-compatible 4-arg constructor (`adoptExisting=false`) so the ~15 existing test/call sites compile unchanged. The `POST /repositories/{repoId}/workspaces` handler passes it through.

### Frontend — one-click button, derived id, parent = main

- `branch-row.component.ts` — a "Create workspace" button rendered only when the row has **no** workspace (`!workspace()`), emitting a `createWorkspace` output. (`branchOff` is now always `secondary`, since "Create workspace" is the primary action for a workspaceless branch.)
- `branch-tree.component.ts` — forwards a `createWorkspace` output (carries the branch name).
- `branch-list.component.ts` — `onCreateWorkspace(branch)` fires the existing `createMutation` directly (no dialog) with `adoptExisting: true`, `parent = mainBranch()`, and an id from `deriveWorkspaceId(branch)` — the server slug rule (`[^A-Za-z0-9_-]` → `-`, ≤64, not dash-leading), de-colliding against existing active workspace ids by suffixing `-2`, `-3`, … The `createMutation` now carries the optional `adoptExisting` flag.

The main branch is excluded automatically: a freshly cloned repo seeds a workspace on its main branch (`createMainWorkspace`), so main's node is workspace-backed and shows no button.

## Tests

- `WorkspaceControllerTest.testCreateWorkspaceAdoptsExistingBranch` — adopting the fixture's existing `feature` branch (parent `master`) succeeds, records a workspace bound to that branch (verified via the list endpoint — the create-response DTO mapper intentionally ignores `branch`), with `parent = master`.
- `WorkspaceControllerTest.testCreateWorkspaceRejectsExistingBranchWhenNotAdopting` — creating a workspace whose branch already exists **without** `adoptExisting` fails (500), guarding the loud-failure contract of the normal "branch off" path.
- Full `domain` + `service` unit suites green (250 in `service`); frontend lint + production build clean.

## Screenshot baseline — regenerated

`branch-tree.browser.spec.ts` renders `master` and `develop` as workspaceless (`data: null`) nodes, so the "Create workspace" button now appears on them. The `__screenshots__/branch-tree.browser.spec.ts/workspace-tree-chromium-linux.png` baseline was **re-recorded on the sanctioned baked `qits/workspace` renderer** (per `docs/epics/qits-build-setup/features/2026-07-13_screenshot-baseline-renderer-baked-into-image.md`) and verified green — no manual follow-up.

It was regenerated by running the visual spec inside a throwaway `qits/workspace:latest` container (the baked renderer: playwright 1.61.0 / Chromium 149 / pinned font stack) with the repo bind-mounted, as the host uid so the PNG lands in the working tree:

```bash
# SRC = the repo's real host path (Docker Desktop resolves -v sources on the daemon host,
#       not in the calling shell; read it from an existing container's mount config).
docker run --rm -u "$(id -u):$(id -g)" -e HOME=/tmp -v "$SRC:/workspace" \
  -w /workspace/service/src/main/webui qits/workspace:latest bash -lc '
    rm -f src/app/ui/components/repository/__screenshots__/branch-tree.browser.spec.ts/workspace-tree-chromium-linux.png
    node node_modules/@angular/cli/bin/ng.js run qits-ui:test-visual \
      --include="src/app/ui/components/repository/branch-tree.browser.spec.ts"  # run 1: writes baseline
    node node_modules/@angular/cli/bin/ng.js run qits-ui:test-visual \
      --include="src/app/ui/components/repository/branch-tree.browser.spec.ts"  # run 2: verifies → passes
  '
```

Note: the currently-running devcontainer image predates the baked-renderer change (`docker/workspace/Dockerfile` is still uncommitted), so it lacks `/opt/ms-playwright` — regenerating there won't work until it's rebuilt on the new image. The standalone `qits/workspace:latest` image already has the baked renderer, which is why the throwaway-container route above works today.
